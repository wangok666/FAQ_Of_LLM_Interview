## 6. 如何多卡部署推理qwen-72B?假设2台机器,每台4卡A100,每卡40G,使用ray

### 1. 核心思路与显存预估

* **并行策略**：使用 Tensor Parallelism (TP) 和 Pipeline Parallelism (PP)。对于 vLLM，通常建议将 `tensor_parallel_size` 设置为总显卡数（即 8）。
* **显存分配**：
* 模型权重：$72B \times 2 \text{ bytes} \approx 144\text{GB}$。
* 平均每张显卡分摊：$144\text{GB} / 8 = 18\text{GB}$。
* 剩余空间（每卡约 22GB）将用于存放 KV Cache，足以支持长文本推理。

### 2. 环境准备

确保两台机器可以互相通过 SSH 免密访问，并且安装了相同版本的依赖：

```bash
pip install vllm ray
```
### 3. 启动 Ray 集群

需要在两台机器上建立 Ray 的主从关系。

**在机器 A (Head Node) 执行：**

```bash
ray start --head --port=6379
```

**在机器 B (Worker Node) 执行：**
（将 `<Head-Node-IP>` 替换为机器 A 的内网 IP）

```bash
ray start --address='<Head-Node-IP>:6379'
```

### 4. 使用 vLLM API Server (推荐)

为了更方便地对接前端或应用，建议启动兼容 OpenAI 接口的服务端：

```bash
# 在 Head Node 执行
python -m vllm.entrypoints.openai.api_server \
    --model Qwen/Qwen2.5-72B-Instruct \
    --tensor-parallel-size 8 \
    --distributed-executor-backend ray \
    --host 0.0.0.0 \
    --port 8000
```

vLLM 会自动检测 Ray 集群并分发模型分片

--distributed-executor-backend ray

可以考虑使用 **GPTQ** 或 **AWQ** 4-bit 量化版本的 Qwen-72B

多机 TP 并行对节点间的网络要求极高。如果两台机器之间不是 InfiniBand (IB) 互联，而是普通的万兆网，推理延迟（Latency）会因为跨机通信而大幅增加。


## 7. 如何多卡部署训练qwen-72B?使用 peft 进行 lora

最优:`Ray + DeepSpeed + PEFT`

核心:
`from ray.train.huggingface.transformers import RayTrainReportCallback, prepare_trainer`
`trainer = prepare_trainer(trainer)`

```json
{
  "fp16": { "enabled": true },
  "zero_optimization": {
    "stage": 3,
    "offload_optimizer": { "device": "cpu" }, 
    "offload_param": { "device": "cpu" },
    "overlap_comm": true,
    "contiguous_gradients": true
  },
  "train_micro_batch_size_per_gpu": 1,
  "gradient_accumulation_steps": 4
}
```

```python
import ray
from ray.train.huggingface.transformers import RayTrainReportCallback, prepare_trainer
from ray.train.torch import TorchTrainer
from peft import LoraConfig, get_peft_model
from transformers import AutoModelForCausalLM, TrainingArguments, Trainer

def train_func():
    # 1. 加载模型
    model = AutoModelForCausalLM.from_pretrained("Qwen/Qwen2.5-72B", trust_remote_code=True)
    
    # 2. 配置 PEFT (LoRA)
    peft_config = LoraConfig(
        r=8, lora_alpha=32, target_modules=["q_proj", "v_proj"], lora_dropout=0.05, task_type="CAUSAL_LM"
    )
    model = get_peft_model(model, peft_config)

    # 3. 设置训练参数
    training_args = TrainingArguments(
        output_dir="./output",
        deepspeed="./ds_config.json", # 关键：调用 DeepSpeed
        per_device_train_batch_size=1,
        learning_rate=1e-4,
        bf16=True # A100 建议开启 bf16
    )

    trainer = Trainer(model=model, args=training_args, ...)
    
    # 4. 适配 Ray
    trainer = prepare_trainer(trainer)
    trainer.train()

# 5. 启动分布式执行器
trainer = TorchTrainer(
    train_func,
    scaling_config=ray.train.ScalingConfig(num_workers=8, use_gpu=True) # 8个worker对应8张卡
)
trainer.fit()
```

ZeRO-3 (Zero Redundancy Optimizer)：它会将模型参数、梯度和优化器状态分散到所有 8 张显存中。


## agent 项目框架怎么设计呢?

#### 总述设计理念
基于状态机的多 Agent 协同架构 核心目标是将复杂的业务逻辑（如司机议价、货主反馈、路由分发）解耦为可观察、可干预的原子节点（Nodes），并通过全局状态（State）进行信息流转。

#### 分层架构

- 编排层 (Orchestration Layer - graph/)
将业务流程抽象为有向无环图（DAG）。例如在 shipper 目录下，我设计了 router_node 进行任务分发，shipper_bargain_node 处理核心议价逻辑。通过 state 定义标准的数据契约，确保不同 Agent（如 Driver Agent 和 Shipper Agent）在对话上下文、意图识别和工具调用结果上保持同步。

- 能力层 (Capability Layer - tools/ & knowledge/)
Agent 的行为边界由 Tools 定义。我实现了工具的校验与干预机制（如 tool_validate_node），在 LLM 生成参数后先进入验证环节，确保货运参数（如价格、距离）符合业务风控，再实际调用后端接口。

- 资源层 (Resource Layer - prompt/)
提示词与代码逻辑分离。针对不同场景（议价、纠错、报告）建立 Prompt 仓库，支持版本管理和动态注入，提高系统的灵活性。

- 评测与进化层 (Evaluation & Flywheel - evaluate/)
项目的核心优势,保证 Agent 的准确性

#### 为什么这么设计？

可解释性与调试：由于采用 Graph 架构，每一轮对话停留在哪个 Node、State 发生了什么变化都是透明的，解决了黑盒问题。

长对话内存管理：在 driver_memory_agent.py 中，我设计了基于业务实体的记忆抽取，避免了 Token 溢出同时保留了关键议价信息。

容错设计：通过 correction_node，当 LLM 输出格式错误或逻辑不通时，系统会自动触发自纠错（Self-Correction）流程，而不是直接崩溃。
